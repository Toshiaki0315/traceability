# Blockchain Traceability Sample (PBFT・コンソーシアム型学習用)

このプロジェクトは、製造・サプライチェーンの工程データ記録を題材に、ブロックチェーンの基本原理と**エンタープライズ向け（コンソーシアム型）ブロックチェーン**の仕組みを学ぶためのPythonによる概念実証（PoC）サンプル実装です。

## アーキテクチャ概要

## 実装済みの機能

### ステップ1: 電子署名によるアイデンティティ管理
データが「誰によって」記録されたかを保証するため、公開鍵暗号（RSA）を導入しています。
- 各参加者（ノード）は専用の鍵ペア（公開鍵・秘密鍵）を持ちます。
- データ送信時に秘密鍵で署名し、受信側で公開鍵を使って改ざん・なりすましを検知します。

### ステップ2: 複数ノードによるP2Pネットワーク
単一の管理者がチェーンを持つのではなく、複数の参加者が独立した台帳（`Node`オブジェクト）を持つ分散型ネットワークをシミュレーションしています。
- 各ノードは「Leader（リーダー）」または「Replica（レプリカ）」の役割を持ちます。
- ノード間でメッセージをブロードキャスト（一斉送信）できます。

### ステップ3: トランザクションの分離とリーダーの役割
データが発生しても即座にブロック化されず、以下のフェーズを踏みます。
- **未承認プール（Mempool）**: 受信したトランザクションは署名検証後、一時的に `pending_transactions` に蓄積されます。
- **ブロックの提案**: リーダーノードのみが未承認トランザクションをまとめて新しいブロック候補を作成し、ネットワーク全体に提案します。

### ステップ4: PBFT（Practical Byzantine Fault Tolerance）合意形成
リーダーが提案したブロック候補に対して、全ノードが多数決（定足数による合意）を行い、合意が取れたものだけが正式にブロックチェーンに書き込まれます。

### ステップ8: DuckDBを用いたオフチェーン・オンチェーン連携（ハイブリッド方式）
ブロックチェーンの書き込みコストや大容量データの課題を克服するため、実データと暗号ハッシュを分離するハイブリッド設計を学びます。
- **オフチェーン (DuckDB)**: 詳細な温度ログや加工の時系列データなど（大容量）を DuckDB に保存します。
- **オンチェーン (PBFT)**: オフチェーンレコードのIDとSHA-256ハッシュ値のみをトランザクション（`OFFCHAIN_ANCHOR`）として合意・記録します。
- **整合性監査 (Integrity Verification)**: オンチェーンのハッシュと、オフチェーンから再計算したハッシュを比較し、データの改ざん（データベースの直接書き換えなど）を自動検知します。

### ステップ11: トランザクションのバルク処理（バッチ処理）の導入
大量のトランザクション発生時に合意形成の回数を抑え、効率的に動作させるためのバッチ処理機構を導入しています。
- **バルク条件の設定**:
  - `BULK_MAX_COUNT`: 未承認プールに溜まったトランザクション数が閾値（デフォルト10件）に達した際にブロック化。
  - `BULK_MAX_WAIT_SECONDS`: トランザクションがプールに入ってから設定時間（デフォルト5秒）が経過した際に、件数が閾値未満であってもブロック化。
- **バックグラウンド監視**: リーダーノードに常駐する監視スレッド（`_batch_watcher`）がスレッドセーフにバルク条件の判定を行い、自動的にブロック提案（PRE_PREPARE）をトリガーします。

```
  リーダー           レプリカA          レプリカB
     │                  │                  │
     │  PRE_PREPARE     │                  │
     │─────────────────►│                  │
     │─────────────────────────────────────►│
     │                  │                  │
     │    PREPARE       │    PREPARE       │
     │◄─────────────────│                  │
     │                  │─────────────────►│
     │◄────────────────────────────────────│
     │                  │◄─────────────────│
     │                  │                  │
     │    COMMIT        │    COMMIT        │
     │─────────────────►│                  │
     │─────────────────────────────────────►│
     │◄─────────────────│                  │
     │◄────────────────────────────────────│
     │                  │─────────────────►│
     │                  │◄─────────────────│
     │                  │                  │
     ★ ブロック確定     ★ ブロック確定     ★ ブロック確定
```

1. **PRE_PREPARE**: リーダーが「このブロックを追加したい」と全ノードに提案。
2. **PREPARE**: 各ノードが提案を検証し「賛成」の投票を全ノードに送信。
3. **COMMIT**: 定足数（`PBFT_QUORUM`）のPREPARE票を確認したノードが「合意成立」を全ノードに宣言。
4. **ブロック確定**: 定足数のCOMMIT票を受け取ったノードが、自身の台帳にブロックを正式追加。

## 製造業DXにおけるユースケース

食品、医薬品、化粧品など、厳格な品質管理基準（HACCPやGMPなど）が求められる製造現場において、本仕組みをIoTエッジデバイスと連携させることで以下のような効果が期待できます。
- **監査対応の迅速化:** 改ざん不可能な温度管理・加工ログによる強固な品質証明。
- **迅速なリコール対応:** 不良品発生時、対象ロットのピンポイントな追跡と影響範囲の特定。

## 実行環境の構築

### 前提条件
- **Python 3.7以上** がインストールされていること。

### 必要なライブラリのインストール
電子署名とオフチェーンデータベース機能、およびWeb UIを利用するため、`rsa`、`duckdb`、`flask` が必要です。以下のコマンドでインストールしてください。
```bash
pip install rsa duckdb flask
```

## 実行方法

### シミュレーションの実行
ターミナルを開き、以下のコマンドを実行してください。
```bash
python traceability.py
```

### ノード REST API の起動（ステップ10）

各参加者ノードを**独立したプロセス・ポート**で起動し、ノード間通信を HTTP/REST に切り替えます。

```bash
pip install -r requirements.txt
chmod +x scripts/start_nodes.sh
./scripts/start_nodes.sh
```

| ノード | 役割 | URL |
|---|---|---|
| 納入業者 | Replica | http://127.0.0.1:5001 |
| 加工工場 | Leader | http://127.0.0.1:5002 |
| 倉庫 | Replica | http://127.0.0.1:5003 |

#### 主な API エンドポイント

| メソッド | パス | 説明 |
|---|---|---|
| `GET` | `/node` | ノード状態（役割・チェーン高さ・未承認TX数） |
| `GET` | `/chain` | 台帳（ブロックチェーン）の照会 |
| `POST` | `/transaction` | トランザクション送信（全ピアへHTTPブロードキャスト） |
| `POST` | `/propose` | ブロック提案（Leader のみ） |
| `POST` | `/audit` | オフチェーン整合性監査 |
| `POST` | `/pbft/pre_prepare` 等 | PBFT フェーズメッセージ（ノード間自動通信） |

#### 利用例

```bash
# 台帳照会
curl -s http://127.0.0.1:5002/chain | python -m json.tool

# トランザクション送信・ブロック提案は traceability.py 内の鍵で署名した JSON が必要です。
# 詳細は test_traceability.py の TestNodeWebAPI を参照してください。
```

単一ノードのみ起動する場合:

```bash
NODE_ID=加工工場 NODE_ROLE=Leader PORT=5002 \
  PEER_URLS=http://127.0.0.1:5001,http://127.0.0.1:5003 \
  python api.py
```

### Web UI ダッシュボードの起動（ステップ9）
Flask Web サーバーを起動することで、ブラウザからPBFTネットワークの状態確認、トランザクション送信、オフチェーン監査デモをインタラクティブに体験できます。

```bash
python app.py
```

起動後、ブラウザで以下にアクセスしてください：
```
http://127.0.0.1:5000
```

#### Web UIでできること
- **ノードモニタリング**: 全ノード（納入業者・加工工場・倉庫）のリアルタイム状態確認
- **通常トランザクション登録**: ロット情報をフォームから送信し、PBFTブロックへ合意確定
- **ハイブリッド連携デモ**: 詳細な加工ログをDuckDB（オフチェーン）へ保存し、ハッシュをPBFTチェーン（オンチェーン）へアンカリング
- **整合性監査**: 「監査」ボタンでオンチェーン・オフチェーンのハッシュを照合（緑：正常 / 赤：改ざん検知）
- **改ざんシミュレーション**: 「改ざん」ボタンでDuckDBデータを直接書き換えた後、監査で警告が表示されることを体験
- **リアルタイムコンソール**: PBFTのPRE_PREPARE/PREPARE/COMMITメッセージを画面右側のターミナルで可視化

### テストの実行
```bash
python -m unittest test_traceability.py
```

### シミュレーションの流れ
1. **Phase 1 - ネットワーク構築**: 「納入業者」「加工工場（Leader）」「倉庫」の3ノードが起動し、相互にピア登録を行います。
2. **Phase 2 - トランザクション送信**: 納入業者が原材料データを秘密鍵で署名し、ネットワークにブロードキャストします。
3. **Phase 3 - PBFT合意形成**: リーダー（加工工場）がブロック候補を作成し、PRE_PREPARE → PREPARE → COMMIT の3段階で全ノードが投票・合意します。
4. **Phase 4 - 結果の検証**: 全ノードの台帳（ブロックチェーン）が一致していることを確認し、合意形成の成功を表示します。
5. **Phase 5 - 台帳の永続化**: 各ノードの台帳データをファイルに保存し、改ざん検証付きで復元します。
6. **Phase 6 - オフチェーン・オンチェーン連携**: DuckDBに大容量詳細ログを保存し、ハッシュのみをPBFT合意形成でオンチェーンへアンカリングします。その後、改ざん検知を含む監査検証を行います。

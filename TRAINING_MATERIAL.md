# ブロックチェーン基礎＆導入判断研修（サプライチェーン事例）

## 1. はじめに
本研修資料は、ITの専門知識がない方でも「ブロックチェーンの仕組み」を理解し、「自社の業務（台帳管理やシステム）にブロックチェーンを導入すべきか？」を判断できるようになることを目的としています。

製造・サプライチェーンの工程記録をシミュレーションしたプログラムを実際に動かしながら、ブロックチェーンの基礎を学んでいきましょう。

---

## 2. ブロックチェーンとは何か？（超基礎編）

ブロックチェーンを一言でいうと、**「みんなで共有・監視することで、絶対にズル（改ざん）ができないデジタル台帳」**です。

### 従来の台帳システムとの違い
* **従来のシステム（中央集権型）：** 会社の中央サーバーや特定の管理部署に「1つの大きな台帳」があり、管理者がすべての書き込みをコントロールします。管理者が不正をしたり、外部からハッキングされたりすると、データが改ざんされるリスクがあります。
* **ブロックチェーン（分散型）：** ネットワークの参加者全員が「同じ台帳のコピー」を持っています。誰かが新しいデータを書き込もうとすると、全員で「そのデータは正しいか？」を確認し合います。

### なぜ「改ざんできない」のか？
1. **デジタルのハンコ（電子署名）：** データには必ず「誰が書いたか」を証明する強力な暗号のハンコが押されます。他人が勝手にハンコを押す（なりすます）ことはできません。
2. **鎖でつなぐ（ハッシュ値）：** データは「ブロック」という箱にまとめられ、前の箱の内容を暗号化して次の箱に繋ぎます。過去のデータを1文字でも書き換えると、それ以降のすべての箱の鍵（暗号）が合わなくなり、すぐに不正がバレる仕組みになっています。

---

## 3. 企業向け（コンソーシアム型）ブロックチェーンとは？

ビットコインのような暗号資産で使われるブロックチェーン（パブリック型）は、「世界中の誰でも参加できる」反面、データの確定に時間がかかり、膨大なコンピューターパワーを消費します。

しかし、企業の業務で使うブロックチェーン（**コンソーシアム型**）は実務に合わせて少し仕組みが異なります。

* **参加者が決まっている：** 「取引先」「自社の各工場」「協力会社」「倉庫」など、身元が分かっている関係者だけがネットワークに参加します。
* **多数決で素早く決める（PBFTという仕組み）：** 暗号計算の競争ではなく、参加者同士で「このデータ、正しいよね？」と多数決（合意形成）を行います。一定数（例えば3分の2）が賛成すれば、すぐに台帳に記録されるため、ビジネスの実用に向いたスピードが出ます。

---

## 4. 体験：製造・サプライチェーンでの活用イメージ

今回の実習では、**「納入業者」→「加工工場」→「倉庫」**というサプライチェーンを想定したプログラムを使用します。

システム内では、以下のようなことが自動で行われています。
1. **「誰が」記録したかの保証：** 納入業者専用の「秘密鍵」で署名し、なりすましを防ぎます。
2. **みんなで多数決をとる：** 加工工場がデータを記録しようとすると、各拠点（ノード）がデータを検証し、「問題なし！」と合意（多数決）が取れたものだけが全員の台帳に書き込まれます。
3. **大容量データの賢い扱い方（ハイブリッド方式）：** 重いデータは通常のデータベース（DuckDB）に保存し、そのデータの「指紋（ハッシュ値）」だけをブロックチェーンに記録します。これにより動作を軽くしつつ、改ざんを完全に防ぎます。

## 5. 実習：スクリプトを動かして仕組みを体感しよう

ここからは、実際に手元のパソコンでプログラムを動かしながら、台帳の動きを確認していきます。
※黒い画面（ターミナル／コマンドプロンプト）を使いますが、記載の通りにコピー＆ペーストすれば大丈夫です。

### 【準備】リポジトリのダウンロードとアプリの起動

1. **プログラムのダウンロード**
   ターミナルを開き、以下のコマンドを1行ずつコピー＆ペーストして `Enter` を押します。
   ```bash
   git clone [https://github.com/Toshiaki0315/traceability.git](https://github.com/Toshiaki0315/traceability.git)
   cd traceability

```

2. **必要な部品（ライブラリ）のインストール**
```bash
pip install -r requirements.txt

```


3. **システムの起動**
```bash
python app.py

```


※起動後、ブラウザを開いて `http://localhost:5000` （またはターミナルに表示されたURL）にアクセスし、操作画面が表示されれば準備完了です。

---

### 【実習1】台帳へのデータ追加と多数決（合意形成）

まずは、データを登録して「みんなで合意して記録する」流れを確認します。

1. **データの入力と追加**
* 画面の入力フォームから、テストデータを入力します。
（例：入力者「納入業者」、内容「部品A（LOT-001）を納品」）
* 「送信（記録）」ボタンを押します。


2. **多数決のログを確認する**
* データ送信直後、起動しているターミナル（黒い画面）を見てください。
* `PRE-PREPARE`（リーダー工場が「このデータを記録しよう」と提案）
* `PREPARE`（他の拠点が「内容とハンコに問題なし」と確認）
* `COMMIT`（みんなの賛成が集まり、正式に記録決定）
* この3ステップを経て、初めて台帳（ブロック）に追加される様子が確認できます。



---

### 【実習2】データの確認と、2種類の「削除」

ブロックチェーンは「一度書いたら消せない」のが基本ですが、業務上どうしても消す必要がある場合の仕組み（ステップ12の機能）を体験します。

1. **データの確認**
* 画面上の「履歴一覧」などを開き、先ほど追加したデータがオンチェーン（指紋）とオフチェーン（実際の文字データ）の両方に記録されていることを確認します。


2. **削除（Soft Delete：論理削除）**
* 対象のデータを「削除」してみます。
* **結果：** 画面上からは見えなくなります（または「削除済み」マークが付きます）が、裏側のデータベースにはデータが残っており、「過去にこのデータが存在し、後から削除された」という事実と履歴が保たれます。一般的な業務エラーの修正に使われます。


3. **完全削除（Hard Delete：物理削除）**
* 対象のデータを「完全削除」してみます。
* **結果：** オフチェーンのデータベース（DuckDB）から文字データそのものが完全に消去されます。
* **ポイント：** データは消えますが、ブロックチェーン上には当時の「指紋（ハッシュ値）」だけが永遠に残ります。これにより、「個人情報などは完全に消しつつ、『過去に確かにその記録が存在した』という証明だけは残す」ことが可能になります。



---

### 【実習3】セキュリティ体験（なりすまし防止・改ざん防止）

ブロックチェーンの強固なセキュリティを、あえて「悪いこと」をして確かめてみます。

1. **なりすまし防止の体験**
* データを登録する際、わざと「違う人の秘密鍵（ハンコ）」を使うか、ハンコのデータを少し書き換えて送信してみてください。
* **結果：** 送信時にエラーとなり、ネットワークから「署名が不正です（偽物のハンコです）」と弾かれます。なりすましは不可能です。


2. **改ざん防止の体験**
* 今回のシステムは、実データを裏側のデータベース（DuckDB）に置いています。
* 別のツール（データベース管理ソフト等）を使って、保存されているDuckDBのデータを直接1文字だけ書き換えて（改ざんして）みてください。
* その後、アプリ画面から「データ監査（検証）」を実行します。
* **結果：** ブロックチェーンに刻まれた「絶対に変えられない指紋（ハッシュ値）」と、改ざんされたデータから計算した「今の指紋」が合わなくなり、画面に **「⚠️データが改ざんされています！」** と強い警告が出ます。

## 6. 自社台帳へ導入すべきか？（導入判断のポイント）

実習で体験した通り、ブロックチェーンは非常に強力ですが、複雑な処理を行っています。自社のシステムに導入すべきか、以下のチェックリストで判断しましょう。

### 🟢 導入に向いているケース（メリットが活きる）

* [ ] **複数の組織や会社間でデータを共有・更新する。**（例：サプライチェーン全体のトレーサビリティ）
* [ ] **互いに完全に信用しきれない、あるいは第三者に対して透明性を証明する必要がある。**（例：産地偽装の防止、監査法人への証明）
* [ ] **過去の履歴が「絶対に改ざんされては困る」性質のものである。**

### 🔴 導入に向いていないケース（従来のデータベースで十分）

* [ ] **自社内の1つの部署（または数人）だけで管理・完結しているデータである。**
* [ ] **データを頻繁に修正したり、完全に消去したりする必要がある。**（実習2の通り、消すための設計と運用が非常に複雑になります）
* [ ] **1秒間に何万件もの膨大なデータを処理する必要がある。**（多数決をとるため、通常のデータベースより動作が遅くなります）

### 💡 結論

ブロックチェーンは「何でもできる魔法のデータベース」ではありません。
「複数人で共有する」「絶対に改ざんされたくない（信頼関係をシステムで担保したい）」という明確な目的がある場合にのみ、導入の価値があります。

---

## 7. まとめ

* ブロックチェーンは「みんなで監視して改ざんを防ぐ共有台帳」です。
* ビジネス用途では、関係者だけで多数決をとる「コンソーシアム型」が適しています。
* 実データは外部DBに置き、指紋だけをブロックチェーンに乗せる（ハイブリッド方式）ことで、大容量データにも対応しつつ改ざんを即座に検知できます。
* 「本当に改ざん防止と、複数組織での共有が必要か？」を基準に、従来のシステムと比較して導入を判断しましょう。
